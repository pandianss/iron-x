If new DS crosses a lockout threshold, set users.locked_until and users.enforcement_mode = 'HARD' 5. Emit a domain event: SCORE_UPDATED (already consumed by frontend WebSocket via DisciplineSubscriber) - Log each recalculation with userId, old score, new score, delta - Handle errors per-user — one failure must not abort the batch Write the complete cron job implementation.

03
Lockout Middleware — Not Live-Enforced
Critical
locked_until and enforcement_mode fields exist on User but policyEnforcementMiddleware does not check them on incoming requests. A locked user can still access the system freely.
Agent Prompt →
TASK: Enforce lockout in policyEnforcementMiddleware. File: backend/src/middleware/policyEnforcementMiddleware.ts Requirements: - After JWT auth resolves userId, query users.locked_until and users.enforcement_mode from Postgres - If locked_until is in the future: Return HTTP 423 Locked with body: { code: 'HARD_LOCKOUT', lockedUntil: ISO_STRING, message: '...' } - If enforcement_mode = 'SOFT': Attach req.restrictedMode = true (downstream handlers check this) - Exempt routes: POST /auth/*, GET /discipline/state, POST /discipline/refresh (user must be able to see their score and acknowledge the lockout) - Cache the user enforcement state for 60s per userId to avoid per-request DB hits (use in-memory Map with TTL) Write the complete updated middleware file.
